# A/B verification protocol

How the extracted C core is proven equivalent to its two references:
the original 6809 source (`disassembly/daggorath/original-source/`) and
the C++ SDL port (`native-port/DungeonsOfDaggorath/`).

## Why there is no wall-clock lockstep A/B against the C++ port

The plan originally called for replaying identical inputs through both
the C++ port and the desktop shim and diffing state dumps at jiffy
checkpoints.  That cannot be byte-exact by construction:

- the port's scheduler runs on SDL millisecond wall-clock with a 17 ms
  "jiffy"; the core runs on true 60 Hz jiffies (16.67 ms).  Every
  time-derived counter (heartbeat, torch burn, creature movement)
  drifts ~2% between the two, and the RNG consumption diverges with it.
- the port's text state is ASCII in the grids; the core keeps original
  5-bit display codes.

Instead, equivalence is layered so that every piece is exact:

## Layer 1 - module-level A/B (`make -C tests ab-modules`)

The port's own C++ code, pasted **verbatim** into
`tools/ab_modules_ref_body.hpp` (only SDL-entangled methods omitted),
is driven through identical operation sequences as our C modules, both
sides CRC-tagging every state byte.  Outputs must be identical:

| section | port code (verbatim)              | our code                     |
|---------|-----------------------------------|------------------------------|
| DUNGEON | Dungeon::DGNGEN/CalcVFI/VFIND/... | core/dungeon.c               |
| PARSER  | Parser::PARSER/EXPAND/GETTOK/...  | core/parser.c (full corpus)  |
| OBJECT  | Object::Reset/CreateAll/OBIRTH/...| core/object.c (all 63 + 8)   |
| CREATUR | Creature::NEWLVL/CBIRTH, SYSTCB   | core/creature.c+sched.c, 5 lvls |
| PLRMATH | Player::DAMAGE + HEARTR formula   | core/player.c (2000+ sweeps) |

Normalization: the port stores frequencies in ms, the core in jiffies;
the ref side CRCs `(ms*6)/100`, which is exact for every DTABAS value.

Layer 1 is the port-parity proof: the game *logic* is byte-identical.
The unit tests (`make -C tests`) additionally pin the RNG stream, the
VECTOR.ASM rasterizer and the radix-7 scaler against ASM-derived
goldens - i.e. against the *original*, where the port itself deviates
(float /127 scaling, gray-line fades).

## Layer 2 - scenario goldens (`make -C tests scenarios`)

End-to-end regression freeze of the whole core through the desktop
shim.  Under `--headless` the shim runs a virtual clock (one jiffy per
`plat_yield`), so a scripted replay is bit-reproducible.  At every
command prompt (`core_prompt_hook`) the shim dumps full game state:
vitals, hands, RNG seed, scheduler table, all objects, all live
creatures, maze CRC, and the text grids decoded to ASCII.

| scenario   | what it locks                                              |
|------------|------------------------------------------------------------|
| s1-demo    | attract mode: demo stream, seer map, creature AI, combat   |
| s2-opening | real game: torch/sword handling, examine, save/load round  |
|            | trip, drop/get, move/turn, every parser error path         |
| s3-combat  | combat soak: hits both ways, heart acceleration, faint,    |
|            | death fade dismissal, restart, creature regeneration       |
| s4-descent | scripted maze route (tools/print_maze.c), CLIMB DOWN a     |
|            | hole, mid-game NEWLVL, inverse-video level rendering       |

Regenerate after an intentional behavior change with
`tests/run-scenarios.sh --regen` and review the golden diff like code.
The scenario scripts are generated by `tests/scenarios/make-keys.py`
(the readable source of truth for the key streams).

These goldens are the contract for Phases 2/3: the Spectrum Next and
MEGA65 backends must reproduce them (same core, same inputs), leaving
only each backend's plat_* layer to verify per platform.

## Layer 3 - coverage (`make -C tests coverage`)

Line coverage of `core/` from the scenario replays alone: **80.3%**
(2683 lines) at the time task 9 closed; the remainder is deep-item
content (rings/flasks/scrolls carried by deep-level creatures) plus
defensive branches.  gcovr does not install on this host, so
`tests/run-coverage.sh` aggregates plain gcov output.

## Backend verification layers (Phases 2-4)

The layers above freeze the core; each backend then proves ITS pieces
against that frozen reference (details and results live in the two
platform-notes docs):

- **ab-mos** (`tests/Makefile`): the whole module A/B harness compiled
  by llvm-mos and run on a simulated 6502 - output must be identical
  to the host build's (MEGA65 codegen proven before the backend).
- **z80draw** (`tests/z80draw/run.sh`): the Next's Z80n asm rasterizer
  vs the normative core/draw_ref.c over a deterministic corpus under
  z88dk-ticks - byte-identical framebuffers.
- **z80scale** (`tests/z80scale/run.sh`): the Next's Covox
  volume-scaling asm + its zsdcc-compiled LUT builder vs a gcc
  reference under z88dk-ticks - byte-identical output for all tiers.
- **Fixed-scene diffs** (`tests/next-scene/`, `tests/mega65-scene/`):
  the same deterministic game start rendered on each emulated target,
  compared byte-for-byte against the desktop shim's framebuffer
  (0 differing bytes on both).

## Bugs the protocol has caught

- `viewer_fade.c`: the death/victory hold-frame loop never ticked the
  clock, so the key that dismisses it could never be seen (hang).
- `sched.c/viewer_fade.c`: key polling was FAINT-gated everywhere, but
  the port's fades poll SDL directly (`Scheduler::keyCheck`) - dying
  while fainted made the death fade permanent.  `sched_kbd_poll()` now
  mirrors the port's behavior.
- (pre-protocol, layer-1 groundwork) CEI_VLA byte 209 vs ROM 210: a
  transcription bug in the 20-year-old port, fixed to match the ROM.
